# PO TODO Page Design

## Goal

Add a formal PO TODO page in the app sidebar for all jobs that require a PO. The page is a batch workbench for the same PO workflow currently handled inside Job Detail > PO.

The page should help staff move PO jobs through:

1. `待发邮件`
2. `等待 PO`
3. `Invoiced`
4. `Completed`

`Completed` jobs no longer appear on the PO TODO page. These are PO-specific workflow states and must not change the normal job status used by the Jobs page.

## Entry Point

Use the existing sidebar `PO` item as the entry point, but upgrade it from the current preview route into the formal PO TODO page.

Recommended route:

- `/po`

The existing `/po-dashboard-preview` page can be replaced or redirected once the formal page is implemented.

## PO Workflow States

The PO TODO page uses a PO-specific state separate from `jobs.status`.

- `待发邮件`: The job needs a PO, but the system has not confirmed the PO request email has been sent.
- `等待 PO`: The PO request email has been sent or manually marked sent, and the job is waiting for a PO number.
- `Invoiced`: A PO number has been confirmed, Xero reference has been updated, and the Gmail thread/message has been labeled `invoiced`.
- `Completed`: Staff marked the invoiced PO job complete. Completed jobs are hidden from the PO TODO page.

State transition:

```text
待发邮件
  -> Gmail sent check confirms sent, or staff manually marks sent
等待 PO
  -> staff confirms a PO number
Invoiced
  -> staff selects row(s) and marks complete
Completed
  -> hidden from PO TODO
```

## Sync Behavior

The page syncs Gmail automatically:

- On page load.
- Every 1 hour while the page remains open.
- When staff clicks a manual `同步 Gmail` action.

Sync scope:

- Include `待发邮件`.
- Include `等待 PO`.
- Exclude `Invoiced`.
- Exclude `Completed`.

The sync checks Gmail Sent for PO request emails and checks replies for detectable PO numbers using the existing Gmail thread sync and PO detection logic.

After each sync, refresh the tab counts and the current table.

## Tabs

### 待发邮件

Shows active need-PO jobs where a PO request has not been confirmed as sent.

Main actions:

- Preview the PO draft email.
- Check or manually mark whether the email was sent.

### 等待 PO

Shows jobs where the PO request has been sent but no PO number has been confirmed.

During Gmail sync, if the system detects a PO number in the supplier reply, it should prefill the PO input. Do not show a visible "system detected" hint in the table. Staff can edit the value or type a PO number manually before confirming.

Main action:

- Enter or review the PO number and click confirm.

### Invoiced

Shows jobs where PO confirmation has completed:

- PO number saved.
- Xero reference updated.
- Gmail thread/message labeled `invoiced`.

This tab is mostly read-only and supports:

- Open Job Detail.
- Open Xero invoice.
- Open Gmail thread when available.
- Select rows with checkboxes.
- Select all visible rows.
- Batch `标记完成`.

`标记完成` is only available in the `Invoiced` tab. Marking complete moves the selected rows to `Completed`, so they disappear from PO TODO.

### 车牌

Clicking the plate opens:

```text
/jobs/{jobId}?tab=PO
```

### Xero

Shows an `Open` action when the job has an external Xero invoice id. It opens the Xero invoice URL.

### PO 草稿

For unsent jobs:

- Show a `预览` button.
- Clicking it opens an in-app modal showing the same draft email content generated by Job Detail > PO.
- The preview should include To, Subject, and body/HTML preview.
- The modal may include an action to create or open the Gmail draft using existing draft behavior.

For sent jobs:

- The table button is disabled or replaced with read-only `已发送`.

The PO TODO page should not create a different email template. It should reuse the Job Detail PO draft source.

### 是否发送

On page load and sync, the system checks Gmail Sent.

If Gmail finds the matching PO request email:

- Display read-only text: `已发送`.
- Persist the sent log if needed.
- Move or keep the job in `等待 PO`.

If Gmail does not find the email:

- Display a clickable button: `已发送`.
- The button means "manually mark as sent".
- On click, show a confirmation dialog: `Gmail 没有找到这封 PO 邮件，确认手工标记为已发送吗？`
- If confirmed, update the PO-specific state to `等待 PO`.
- Record that the sent state source is manual.
- Move the row out of `待发邮件`.

The UI should make the read-only text and clickable button visually distinct even though both display `已发送`.

### PO

Shows an input plus confirm button.

In `等待 PO`:

- If sync detected a PO number, prefill the input.
- If no PO is detected, leave it empty.
- Staff can type or edit a PO number.
- Clicking confirm starts the PO confirmation workflow.

In `Invoiced`:

- Show the confirmed PO number read-only.

## Confirm PO Workflow

The frontend sends only the PO number entered by staff:

```json
{
  "poNumber": "12345"
}
```

The backend derives the updated Xero reference.

Reference update rule:

- Replace the `Pending` or `pending` token in the current Xero reference with the input PO number.
- If there is no pending token, prepend the PO number using the existing Job Detail PO fallback behavior.

Examples:

```text
Before: PO Pending ABC123
Input: 12345
After: PO 12345 ABC123
```

```text
Before: pending ABC123 repair
Input: 12345
After: 12345 ABC123 repair
```

The confirm workflow should run these steps:

1. Save PO number locally.
2. Update Xero invoice reference.
3. Add Gmail label `invoiced`.
4. Update PO-specific state to `Invoiced`.

The frontend must show a progress dialog similar to the delete job flow. Each step should show:

- pending
- running
- success
- failed
- message

The final response should include step results whether the workflow fully succeeds or partially fails. This lets staff see exactly where the process stopped.

Recommended failure behavior:

- If saving the PO number fails, stop before Xero/Gmail work.
- If Xero reference update fails, do not mark the PO state `Invoiced`.
- If Gmail label fails after Xero succeeds, show partial completion and keep enough information for retry. The UI should not hide the failure behind a generic toast.
- If all steps succeed, set PO state to `Invoiced`.

## Gmail Label Behavior

Gmail label name:

```text
invoiced
```

The label already exists. The system does not create it.

Implementation still needs to resolve the Gmail `labelId` from the existing Gmail labels list, then call Gmail modify on the relevant message or thread.

Preferred target:

- Label the Gmail thread when a thread id is available.
- Fall back to labeling the sent PO request message when no thread id is available.

This requires Gmail scope:

```text
https://www.googleapis.com/auth/gmail.modify
```

If the current Gmail token does not include modify permission, the confirm PO progress dialog should show the Gmail label step as failed with a reconnect/re-consent message.

## Backend API Shape

Recommended API endpoints:

```text
GET  /api/po/todo?status=pendingSend|awaitingPo|invoiced
POST /api/po/todo/sync
GET  /api/po/jobs/{jobId}/draft-preview
POST /api/po/jobs/{jobId}/manual-confirm-sent
POST /api/po/jobs/{jobId}/confirm-po
POST /api/po/jobs/complete
```

`POST /api/po/todo/sync`:

- Syncs only `待发邮件` and `等待 PO`.
- Checks Gmail Sent.
- Checks Gmail replies and PO detections.
- Refreshes PO states.
- Returns counts for checked jobs, found sent emails, found replies, detected PO numbers, warnings, and failures.

`GET /api/po/jobs/{jobId}/draft-preview`:

- Returns the same To, Subject, and HTML/body content used by Job Detail > PO.
- Returns draft state where useful.

`POST /api/po/jobs/{jobId}/manual-confirm-sent`:

- Manually moves a job to `等待 PO`.
- Records source as manual.

`POST /api/po/jobs/{jobId}/confirm-po`:

Payload:

```json
{
  "poNumber": "12345"
}
```

Response should include step status objects:

```json
{
  "success": true,
  "jobId": 123,
  "poNumber": "12345",
  "invoiceReference": "PO 12345 ABC123",
  "steps": {
    "savePo": { "status": "success", "message": "PO saved." },
    "xero": { "status": "success", "message": "Xero reference updated." },
    "gmail": { "status": "success", "message": "Gmail label added." },
    "poState": { "status": "success", "message": "PO state updated." }
  }
}
```

`POST /api/po/jobs/complete`:

Payload:

```json
{
  "jobIds": [123, 456]
}
```

Only jobs currently in `Invoiced` should be accepted. Other states should be rejected or skipped with a per-row result.

## Data Model Notes

Reuse existing `JobPoState` where possible, because it already stores:

- Job id.
- Correlation id.
- Gmail draft id.
- status.
- sent timestamps.
- reply timestamps.
- detected PO number.
- confirmed PO number.

The existing enum may need one additional effective state or mapping for:

- `Completed`

If `PoConfirmed` currently means "PO confirmed", it can map to the page's `Invoiced` state only after the Xero reference and Gmail label steps succeed. If that is not guaranteed today, add explicit metadata or a status value so the page can distinguish "PO number saved" from "fully invoiced workflow done".

Manual sent state should be auditable. Either:

- Add a manual sent log entry to `GmailMessageLogs` with an explicit manual/source marker, or
- Add a source field to PO state.

Prefer preserving the distinction between Gmail-confirmed sent and manually-confirmed sent for debugging.

## Testing

Backend tests:

- Gmail sent check moves pending-send jobs to awaiting-PO when a sent message exists.
- Manual sent confirmation moves a job to awaiting-PO and records manual source.
- Gmail reply PO detection prefills the PO field in the waiting tab response.
- Confirm PO replaces `Pending` or `pending` in the Xero reference.
- Confirm PO returns step results for success and partial failure.
- Gmail label step fails clearly when `gmail.modify` is missing.
- Batch complete only accepts invoiced rows.

Frontend tests:

- Page load calls sync then loads the selected tab.
- Sent column renders read-only text when Gmail confirmed sent.
- Sent column renders a clickable button when Gmail did not find sent mail.
- Waiting tab prefills detected PO without showing a detection hint.
- Confirm PO progress dialog displays per-step statuses.
- Invoiced tab supports select all and batch complete.

## Out of Scope

- Changing the normal job status.
- Creating Gmail labels.
- Syncing completed jobs.
- Designing a new PO email template separate from Job Detail > PO.
